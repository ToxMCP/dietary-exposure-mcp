# Release Checklist

Use this checklist before tagging or publishing Dietary Exposure MCP.

The complete local gate is available as one command and does not require
GitHub Actions:

```bash
./scripts/verify_release.sh
```

It runs the checks below, audits the locked runtime dependencies, scans for
secrets, creates an SBOM and checksums, and smoke-installs the built wheel in a
clean environment. Generated release binaries remain under the ignored
`artifacts/releases/` directory.

For a public release, also run:

```bash
uv run python scripts/public_release_audit.py
uv run python scripts/public_release_audit.py --history
```

The current-tree audit is blocking. Historical metadata must be sanitized by a
coordinated rewrite or explicitly accepted by the repository owner before the
visibility change.

## Scope and claim review

- Confirm the README and boundary docs still describe Dietary MCP as a regulatory-support module rather than a final decision engine.
- Confirm new workflows do not imply direct-use oral ownership, PBPK execution, or unsupported submission equivalence.
- Confirm any new family or evidence layer has matching readiness and submission posture.

## Scientific-content review

- Confirm the current OpenFoodTox source and any bundled snapshot remain separate, version-pinned records.
- Review any source-currency warnings and the OpenFoodTox reconciliation status before release.
- Review newly added source records for official provenance and correct jurisdiction metadata.
- Review new reference values, method records, and evidence objects for conflict preservation and source linkage.
- Confirm new review-only content is not accidentally marked `allowed`.
- Confirm notes and docs do not overstate the maturity of emerging-contaminant support.

## Artifact generation

Run:

```bash
uv run dietary-mcp-generate-artifacts
uv run dietary-mcp-validate
uv build
uv run python -m dietary_mcp.release_artifacts
```

Review:

- `docs/releases/v0.1.0.release_metadata.json`
- `docs/releases/v0.1.0.validation_dossier.json`
- `docs/releases/v0.1.0.readiness_report.json`
- `docs/releases/v0.1.0.downstream_dry_runs.json`

## Verification gates

Run:

```bash
uv run ruff check src/dietary_mcp tests
uv run python -m pytest
```

Check:

- release counters moved as expected
- scientific-review dossier status remains `draft_ready` while signoff is pending
- no packaged-asset parity regressions
- no broken schema/example manifests

## Repo hygiene

- Confirm `README.md`, `CONTRIBUTING.md`, and `docs/release_checklist.md` still match the current repo surface.
- Confirm `CITATION.cff`, `THIRD_PARTY_NOTICES.md`, `SECURITY.md`, and the selected release note are current.
- Confirm `.gitignore` excludes local caches, builds, and task exports.
- Confirm no private local task files or transient artifacts are staged for push.
- Confirm no tracked file contains personal workstation paths, local file URIs, or personal email addresses.

## Public GitHub software release

- Review all issues, pull requests, comments, and attachments before changing visibility.
- Resolve or explicitly accept historical personal emails and workstation paths.
- Enable GitHub private vulnerability reporting.
- Confirm repository secrets, Actions permissions, default branch, and branch rules are suitable for public visibility.
- Confirm the README, limitations, and release note label `v0.1.0` as screening only.
- Keep `0.1.0` off PyPI unless a separate package-index publication is reviewed and approved.
- Attach the wheel, sdist, SBOM, and `SHA256SUMS` generated by `verify_release.sh`.
- Keep public operation on local stdio; a hosted HTTP endpoint requires a separate deployment-security review.

## Scientific promotion follow-up

- Keep all OpenFoodTox bulk records `review_required` until qualified review changes that state.
- Record positive independent toxicologist or dietary-risk-assessor signoff before claiming scientific validation.
- Record qualified human approval of the remediated OpenFoodTox 3.0 version 1.2
  high-impact report and its exact canonical hash under ADR 0007. The prior
  `not approved` disposition remains part of the review history.
- Resolve the intended-distribution rights posture before downstream redistribution of WHO-derived profiles.
- Recheck current primary authority sources before any decision-relevant use.
